# Design — Trilha B 5.1: Agenda de Reuniões (schema + CRUD + tela)

> **Status:** aprovado no brainstorming (2026-07-09). Pronto para virar plano de implementação.
> **Escopo:** apenas a **FASE 5.1** da Trilha B. As fases 5.2 (capacidade/horários) e 5.3 (ações/vínculo/`create_meeting_task`) ficam fora — 5.2 depende do `businessHours.ts` da Trilha A.
> **Natureza:** greenfield — verificado no banco `tsltxvswzdnlmvljpryh` que **não existe** tabela de reunião/agenda.

---

## 1. Objetivo e contexto

Criar o módulo **Agenda de Reuniões**: a entidade `meetings` de 1ª classe (hoje reunião/audiência existem só como `pendencia_tipo` em `user_tasks`), com CRUD gated por papel, histórico automático e uma tela de listagem/detalhe. Já deixa prontos os **ganchos da Trilha D** (colunas Google + botão de sincronizar desabilitado), sem qualquer chamada real à API do Google.

Esta 5.1 é a **dependência** que destrava a Trilha D (sync Google Agenda), que estava bloqueada por não haver tabela onde ancorar `google_event_id`.

### Decisões travadas (do briefing, não reabrir)
- Nome da tabela: **`meetings`**; **colunas em inglês** (consistência com `clients`/`user_tasks`); **rótulos na tela em PT-BR**.
- Migration **aditiva** via `apply_migration`; **nunca `db push`** (dispara `drop_plaintext_pii`). Não duplicar registro em `schema_migrations`.
- **RLS 100%** nas tabelas novas; RPCs de escrita com **gating de papel por dentro**; toda nova `SECURITY DEFINER` entra na lista do R-1 (Hardening).
- Enum de status em inglês, rótulo PT-BR na tela.
- A criação de tarefa a partir do agendamento (5.3) usará **`create_meeting_task`** (RPC nova dedicada) — **não** `create_user_task` (admin-only, intocável). Fora do escopo da 5.1, apenas registrado.

### Decisão de brainstorming (2026-07-09)
- **Estagiária de recepção (`estagiaria_recepcao`, Yasmin) = recepção plena** na Agenda (lê e escreve). Alinha com a policy de pendências (que já a inclui) e diverge propositalmente do helper `is_recepcao_or_socio()` (que a exclui). → criar helper dedicado `meetings_can_access()`.

## 3. Acesso / RLS (obrigatório, 100%)

### 3.1. Helper `public.meetings_can_access()`
`RETURNS boolean LANGUAGE sql STABLE SECURITY DEFINER SET search_path = public`. Retorna `true` para o usuário atual quando:
- `role_template.code` ∈ (`socio`, `lider_recepcao`, `recepcionista`, `estagiaria_recepcao`), **ou** `code LIKE 'adv_%'`; **ou**
- `public.is_master_admin(auth.uid())` **ou** `public.has_role(auth.uid(), 'admin')`.

(Segue o formato de `is_recepcao_or_socio()`/`is_socio_or_advogado()`, com a estagiária incluída.)

### 3.2. Policies
- `public.meetings`: `ENABLE ROW LEVEL SECURITY`; **policy SELECT** `TO authenticated USING (public.meetings_can_access())`. **Nenhuma** policy de INSERT/UPDATE/DELETE — toda escrita passa pelas RPCs `SECURITY DEFINER` (que furam o RLS), padrão idêntico ao kanban (`task_time_entries`).
- `public.meeting_audit_log`: `ENABLE ROW LEVEL SECURITY`; **policy SELECT** `USING (public.meetings_can_access())`.

### 3.3. Verificação de segurança
Rodar `get_advisors(security)` após a migration — **sem** alerta de "RLS ausente" nas duas tabelas novas (aceite 5.1 #3).

## 4. Superfície de RPCs (5.1)

Todas: `SECURITY DEFINER`, `SET search_path = public`, `REVOKE ALL ... FROM PUBLIC` + `GRANT EXECUTE ... TO authenticated`, e **gate interno** `IF NOT public.meetings_can_access() THEN RAISE EXCEPTION`. Todas entram na **lista do R-1 (Hardening)**.

| RPC | assinatura (resumo) | comportamento |
|---|---|---|
| `create_meeting` | campos do agendamento | valida gate; `created_by = auth.uid()`; `end_time := coalesce(p_end_time, p_start_time + interval '15 min')`; retorna `id`. |
| `update_meeting` | `p_id` + campos editáveis (incl. `status`) | valida gate; atualiza campos informados; `updated_at = now()`. As mudanças relevantes viram histórico via trigger. |
| `delete_meeting` | `p_id` | **hard delete restrito a sócio/admin** (`is_socio() OR kanban_can_admin(auth.uid())`) — correção de engano. Fluxo normal de cancelamento é por `status`, não delete. |
| `get_meeting_audit` | `p_meeting_id` | espelha `get_task_audit`: valida gate, retorna histórico com nome do ator via `profiles`, `LIMIT 200`. |

**Leitura de lista/detalhe:** `supabase.from('meetings').select(...)` direto no cliente (RLS cuida da visibilidade), como em Clientes/Kanban — sem RPC de leitura.

> **Fora da 5.1** (registrados p/ 5.2/5.3): enforcement de capacidade (≤2 por slot) na RPC de criação/reagendamento; `create_meeting_task`; ações de abrir cadastro do cliente.

## 5. Frontend

- **Página** `src/pages/Agenda.tsx` (lazy em `App.tsx`), **rota `/sistema/agenda`** (padrão `/sistema/kanban`).
- **Menu**: adicionar a chave `"agenda"` ao `roleVisibility.menuItems` dos papéis `admin, lawyer, receptionist, intern` — que é exatamente o conjunto que o gate `meetings_can_access()` autoriza, conforme o mapeamento canônico `role_template.code → app_role` de `invite-employee` (socio→admin, adv_*→lawyer, lider_recepcao/recepcionista→receptionist, estagiária→intern). **Não** incluir `director`/`manager` (não há usuários nesses papéis e o banco não os autoriza — evita menu morto). Montar o `MenuItem` correspondente no shell (`JurisCloudOS`) com ícone de calendário e `action` navegando para `/sistema/agenda`.
- **Listagem**: visão por **dia/semana**, filtros por **advogado** e **status** (rótulos PT-BR do enum). Leitura via `supabase.from('meetings')`.
- **Detalhe/edição**: modal ou painel no padrão do `TaskDetailModal`, com formulário dos campos e **seção de histórico** consumindo `get_meeting_audit`. CRUD chamando `create_meeting`/`update_meeting`/`delete_meeting`.
- **Gancho Trilha D**: botão **"Sincronizar com Google Agenda" desabilitado** + tooltip "requer conexão Google (em breve)". Nenhuma chamada real.
- **Tipos**: `type` é dropdown sugerido — *Consulta inicial · Retorno · Assinatura · Acompanhamento · Cooperativa · Outro* — persistido como texto livre.

### Observação sobre `app_role` × `role_template.code`
O frontend gate-eia menu por **`app_role`** (`admin, director, manager, lawyer, receptionist, intern, tech...`), taxonomia distinta dos `role_template.code` do RLS (`socio, adv_*, recepcionista...`). O RLS (fonte da verdade) usa os códigos; o menu usa `app_role`. Ambos precisam liberar o mesmo conjunto de pessoas — recepção + advogados + gestores + admin.

## 6. Migration & verificação

1. `apply_migration` **aditivo e idempotente**: enum via `DO $$ ... EXCEPTION WHEN duplicate_object`; tabelas com `CREATE TABLE IF NOT EXISTS`; policies com `DROP POLICY IF EXISTS` + `CREATE POLICY`; funções `CREATE OR REPLACE`; trigger com `DROP TRIGGER IF EXISTS` + `CREATE TRIGGER`.
2. **Nunca `db push`** (R-2).
3. Pós-DDL: `execute_sql` conferindo existência de tabela/enum/policies/trigger + `get_advisors(security)` sem alerta de RLS ausente.
4. `types.ts`: regeneração roda no CI/pelo Ryan (sem Node local). Se a tela precisar compilar antes, ajustar o tipo de `meetings`/`meeting_status` manualmente em `src/integrations/supabase/types.ts`.
5. Deploy de edge: **não há** edge nesta fase. (Se surgir, é deploy manual do Ryan com CI `edge` verde.)

---

## 7. Critérios de aceite (5.1)

1. Tabela `meetings` + enum `meeting_status` + `meeting_audit_log` + trigger criados por migration **aditiva**, **RLS ativa**, **sem `db push`**.
2. Novo item de menu "Agenda"; criar/ler/editar/excluir reunião com **todos os campos** funcionando na tela.
3. RLS restringe aos papéis certos (recepção incl. estagiária + advogados + gestores + admin); `get_advisors(security)` **sem** alerta de RLS ausente.
4. Ganchos da Trilha D presentes: colunas Google criadas + botão "Sincronizar" **desabilitado**; sync real adiado.
5. Histórico da reunião registrado (mudança de status/data/horário/advogado aparece via `get_meeting_audit`).

## 9. Riscos / atenção

- **Desync repo↔banco**: a migration entra no banco via `apply_migration`; garantir que o arquivo `.sql` correspondente exista em `supabase/migrations/` com o mesmo timestamp, sem duplicar linha em `schema_migrations`.
- **`get_advisors` limpo** é gate de aceite — não fechar a fase com alerta de RLS.
- **R-1 Hardening**: registrar `create_meeting`, `update_meeting`, `delete_meeting`, `get_meeting_audit` e o helper na lista de funções `SECURITY DEFINER` auditadas.
