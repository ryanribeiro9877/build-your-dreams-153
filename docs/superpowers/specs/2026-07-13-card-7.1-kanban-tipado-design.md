# Card 7.1 · Kanban por tipo de ação — Design

**Data:** 2026-07-13 · **Onda:** 1 (worktree `wt-71-kanban`) · **Status:** BLOQUEADO na lista validada (Ryan + Rodrigo) — artefatos preparados, **não aplicados**.

## 1. Objetivo
Provisionar **uma board de Kanban por tipo de ação** (jurídica), tipando as boards para habilitar o roteamento automático do 7.2. O motor de Kanban **já existe e está em produção** — 7.1 **não constrói Kanban**; ele (a) adiciona a dimensão "tipo" à board, (b) semeia as boards por tipo, (c) valida "múltiplas entradas por cliente".

## 3. Decisões

| # | Decisão | Racional |
|---|---|---|
| D-1 | **Adicionar `tipo_acao text` (nullable)** a `kanban_boards`, não enum. | A lista validada pode **exceder** `legal_area` (ex.: **Trabalhista** não existe no enum). Texto evita uma migration de enum e mantém flexibilidade. Boards operacionais (ex.: pendências) ficam com `tipo_acao = NULL`. |
| D-2 | **Índice único parcial** `UNIQUE (tipo_acao) WHERE tipo_acao IS NOT NULL`. | Garante **uma board por tipo** (idempotência do seed; base sã para o roteamento do 7.2). |
| D-3 | Boards de tipo são **compartilhadas** (`is_private = false`), não privadas como a board-semente atual. | São quadros operacionais do escritório, não privados de um usuário. Visibilidade por `kanban_can_access_board`. |
| D-4 | Seed via **script `.mjs` idempotente** (`scripts/seed-kanban-type-boards.mjs`), com **inserts diretos por service role** (não pelas RPCs). | As RPCs (`kanban_create_board`/`kanban_set_columns`) exigem `kanban_can_admin(auth.uid())`; sob service role `auth.uid()` é NULL → gate falha. Insert direto (bypass RLS, service role) replica fielmente o template verificado. Alternativa equivalente em SQL via MCP documentada em §7. |
| D-5 | **Não** aplicar migration nem rodar o seed **até a lista fechar**. Evita desync repo↔banco e boards órfãs. | Regra de operação da Onda 1: aditivo aplicado só após a lista; migration entra como espelho versionado **depois** de aplicada. |
| D-6 | 7.2 (roteamento) fará `board.tipo_acao = ut.area::text`. **Ressalva:** uma `user_task` de Trabalhista teria `area = NULL` (não há `trabalhista` em `legal_area`) → não casaria por `area`. | Sinalizado ao 7.2; se Trabalhista virar board, o roteamento precisará de uma chave adicional (tag/department), fora do escopo do 7.1. |

**Fora de escopo (7.2+):** roteamento/distribuição automática, gating de papel por board, prazos/responsáveis/alertas por Kanban, tarefa de revisão humana. Visão consolidada "tarefas do cliente por board" na ficha (achado #8) é **melhoria de front sugerida**, não obrigatória para o 7.1.

## 5. Migration aditiva (PRONTA — aplicar só após a lista)

Aplicar via MCP (`apply_migration`) e versionar o espelho em `supabase/migrations/`. **Ainda não aplicada** (D-5).

```sql
-- 7.1: dimensão de tipo de ação na board de Kanban (aditivo, nullable)
ALTER TABLE public.kanban_boards
  ADD COLUMN IF NOT EXISTS tipo_acao text;

COMMENT ON COLUMN public.kanban_boards.tipo_acao IS
  'Tipo de ação jurídica da board (7.1). Texto para admitir valores fora do enum legal_area (ex.: trabalhista). NULL = board operacional sem tipo. Base do roteamento do 7.2.';

-- uma board por tipo (idempotência do seed + base do roteamento)
CREATE UNIQUE INDEX IF NOT EXISTS kanban_boards_tipo_acao_uidx
  ON public.kanban_boards (tipo_acao)
  WHERE tipo_acao IS NOT NULL;
```

*(Nome sugerido do arquivo-espelho: `supabase/migrations/<timestamp>_kanban_boards_tipo_acao.sql`.)*

## 6. Plano de validação CORRIGIDO (por MCP, após seed)

O briefing pedia "uma user_task em 2 boards = 2 placements" — **impossível** por `UNIQUE(user_task_id)` (achado #7). A prova correta de "múltiplas entradas por cliente" é **N tarefas distintas do mesmo cliente em N boards de tipo diferentes**:

1. **Boards tipadas:** `SELECT name, tipo_acao FROM kanban_boards WHERE tipo_acao IS NOT NULL` → N linhas, uma por tipo da lista, cada uma com 5 colunas na ordem `pendente…cancelado`.
2. **Múltiplas entradas por cliente:** escolher um cliente de teste; criar 2 `user_tasks` de áreas diferentes (ex.: uma `bancario`, uma `previdenciario`); colocar cada uma na board do seu tipo (`kanban_add_task_to_board`); confirmar **2 placements em 2 boards distintas** para o **mesmo cliente**. Depois **remover** as tarefas de teste.
3. **Ficha do cliente:** confirmar que as tarefas do cliente aparecem (achado #8: hoje sem board/coluna — decidir se expor board na ficha é escopo agora ou melhoria posterior).

## 7. Seed — equivalente SQL (opção MCP)
Caso se prefira semear via MCP em vez do `.mjs` (D-4), o padrão por tipo é (idempotente):

```sql
-- exemplo para um tipo; repetir por tipo da lista. Owner = sócio (ex.: dono da board existente).
with novo as (
  insert into public.kanban_boards (name, owner_user_id, is_private, tipo_acao, sort_order)
  select 'Ações Previdenciárias', b.owner_user_id, false, 'previdenciario', 110
  from public.kanban_boards b where b.tipo_acao is null and b.name='Ações Bancárias' -- fonte do owner
  on conflict (tipo_acao) where tipo_acao is not null do nothing
  returning id
)
insert into public.kanban_columns (board_id, name, situacao, position)
select n.id, c.name, c.situacao::task_situacao, c.position
from novo n
cross join (values
  ('Pendente','pendente',0),
  ('Em execução','em_execucao',1),
  ('Concluída (sucesso)','concluida_sucesso',2),
  ('Concluída (sem sucesso)','concluida_sem_sucesso',3),
  ('Cancelada','cancelado',4)
) as c(name,situacao,position);

-- a board bancária existente só recebe o tipo:
update public.kanban_boards set tipo_acao='bancario'
where name='Ações Bancárias' and tipo_acao is null;
```

## 8. Aceite
1. `kanban_boards.tipo_acao` existe (nullable) + índice único parcial.
2. Uma board por tipo da lista validada, cada uma com o template de 5 colunas na ordem correta; board bancária existente tipada como `bancario` (sem duplicar).
3. Seed idempotente (rodar 2× não cria duplicatas).
4. Múltiplas entradas por cliente provadas conforme §6 (N tarefas / N boards), depois limpas.
5. Decisão de Trabalhista registrada.
