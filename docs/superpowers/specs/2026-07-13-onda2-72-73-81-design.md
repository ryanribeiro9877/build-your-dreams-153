# ONDA 2 — Cards 7.2 · 7.3 · 8.1 — Design

Data: 2026-07-13
Base: 7.1 (boards tipadas via `kanban_boards.tipo_acao_id` → `tipos_acao`; card = `user_task`
placada com `UNIQUE(user_task_id)`; motor `kanban_*`; enum `task_situacao`).

## Contexto verificado no banco (não presumido)

- `processes` **não tem** `tipo_acao_id` (nada roteado carrega tipo). → chave a introduzir.
- 13 boards: 12 tipadas + "Ações Bancárias" legada (`tipo_acao_id` null).
- Pendência mora em `user_tasks`: `is_pendencia`, `pendencia_estado`, `pendencia_tipo`,
  `data_fatal`, `process_id`, `client_id`.
- `kanban_move_card(p_task_id, p_column_id, p_position)` — **sem** gate de pendência; já tem
  padrão de gate (bloqueia `concluida_sucesso` quando `task_type.requires_validation` e não validado).
- `kanban_columns(board_id, name, situacao, position)` — coluna inicial = menor `position`.
- `user_tasks` NOT NULL sem default: `task_type_id`, `title`, `assigner_user_id`.
  Defaults: `status='assigned'`, `situacao='pendente'`, `priority='medium'`.
- `task_types` tem `confeccionar_peca_<area>` (stage `confeccao`) por área.
- `client_documents.document_type` = **text com CHECK** (18 valores); tabela chaveada só por
  `client_id` (sem `task_id`/`process_id`). `origem` aceita `sistema`.
- `distribuir_caso` **não existe**.

## Decisões técnicas adotadas (aprovadas por Ryan)

1. **Quem seta `tipo_acao` (7.2):** ambos (IA sugere no intake, recepção confirma). Nesta onda a
   camada de banco só cria a coluna + a RPC; sugestão-IA e confirm-recepção são fiação de front/edge.
2. **"Kanban de inicial" (8.1):** etapa no board do tipo (não board dedicado). DB do 8.1 é mínimo
   (tipo de documento + vínculo minuta↔card); "ligar o pipeline" é front/edge.

## Escopo por camada

### 🟢 Banco (aplicado via MCP `apply_migration` + espelho de migration no repo)

**7.2 — chave de roteamento + distribuição + gating**
- `alter table processes add column tipo_acao_id uuid references tipos_acao(id)` + índice.
- `alter table tipos_acao add column default_task_type_id uuid references task_types(id)` — alvo do
  roteamento (qual `task_type` o card distribuído recebe); nullable, preenchido depois (front/seed).
- `distribuir_caso(p_process_id uuid, p_tipo_acao_id uuid default null, p_task_type_id uuid default null, p_title text default null)`
  `SECURITY DEFINER`:
  1. resolve tipo = `coalesce(p_tipo_acao_id, processes.tipo_acao_id)`; null → exceção.
  2. grava o tipo em `processes` se veio por param.
  3. resolve `task_type` = `coalesce(p_task_type_id, tipos_acao.default_task_type_id)`; null → exceção clara.
  4. acha board com `tipo_acao_id = tipo`; não achar → exceção.
  5. acha coluna inicial (menor `position`) do board.
  6. cria `user_task` (`process_id`/`client_id` herdados do processo, `assigner_user_id=auth.uid()`,
     `title`, `situacao='pendente'`, `status='assigned'`) e placa via `kanban_card_placements`
     (`ON CONFLICT (user_task_id)`), respeitando o gating.
  7. retorna `task_id`.
  - **Passo §24.1 fica como `TODO` explícito** (bloqueado no Rodrigo).
- **Gate de pendência** no topo de `kanban_move_card` (antes do INSERT de placement): se o card **não**
  é pendência (`is_pendencia=false`), o alvo é avanço (`em_execucao`/`concluida_sucesso`) e é mudança
  de situação, e o `process_id` do card tem **outra** `user_task` com `is_pendencia=true` e
  `pendencia_estado='aberta'` → `RAISE EXCEPTION`. Mover a própria pendência, recuar, `concluida_sem_sucesso`
  ou `cancelado` não são bloqueados.

**8.1 — inicial no Kanban (DB-side)**
- Amplia o CHECK de `client_documents.document_type` com `peticao_inicial` e `minuta`.
- `alter table client_documents add column task_id uuid references user_tasks(id)` (nullable) — vincula
  a minuta ao card.

**7.3 — scaffold de config (inerte até Rodrigo)**

  criticidade jsonb, created_at, updated_at)` + RLS (leitura para quem acessa o board; escrita admin).
  Sem valores agora.
