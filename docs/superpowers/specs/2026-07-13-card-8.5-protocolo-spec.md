# Card 8.5 · Etapa de PROTOCOLO — Especificação (SÓ SPEC)

**Data:** 2026-07-13 · **Natureza:** documento de especificação (não há código nesta entrega) · **Onda:** 1 (worktree `wt-85-protocolo-spec`)

> **Aviso de escopo.** Este documento **especifica o contrato** da etapa de protocolo. A automação de protocolar no sistema do tribunal (RPA/integração) é dependência de **outro dono** e está **fora de escopo desta fase** — ver §7. Nada aqui gera migration, edge function ou front.

---

## 1. Objetivo e escopo

### 1.1 O que É
Definir, para a etapa `stage = 'protocolo'` do fluxo processual, o **contrato de entrada/saída**, o **catálogo de modos de falha do sistema externo** (tribunal), os **requisitos do LLM** que assiste a etapa e os **requisitos de auditoria/log**. É a base contratual sobre a qual a automação (V2, outro dono) e o assistente de protocolo serão construídos.

### 1.2 O que NÃO É
- **Não** implementa protocolo automático em PJe/PROJUDI/eproc/ProJuris (isso é RPA/integração — §7).
- **Não** cria schema, RPC, edge function ou tela.
- **Não** altera o gate documental nem o fluxo de validação existentes — apenas os referencia como pré-condições.
- **Não** substitui o protocolador humano nesta fase; hoje o protocolo é **ação humana** (verificado: só existem agentes-placeholder e campos de RPA, sem lógica de integração real).

### 1.3 Fatos verificados que ancoram a spec
Verificado em produção (banco `tsltxvswzdnlmvljpryh`) e no código em 2026-07-13:
- `org_stage` contém `protocolo` (enum canônico em `supabase/migrations/20260527120000_v14_lexforce_org_model.sql:41-62`).
- Fluxo linear: **`revisao → protocolo → audiencia`** (`kanban_next_stage`, `20260615120000_kanban_fase1.sql:19-20`).
- Papel dono da etapa: **`adv_protocolo`** (`kanban_stage_owner_role`, `20260615120000_kanban_fase1.sql:44`).
- `task_types` de protocolo já seedados, todos `area = NULL` (multi-área) e `requires_validation = false`:
  - `protocolar_peca` — "Protocolar peça" — SLA 8h
  - `emendar_inicial` — "Atender emenda à inicial"
  - `protocolar_acordo` — "Protocolar acordo aceito" — SLA 8h ("mesmo dia")
- Cada `user_task` pode referenciar um cartão de kanban externo via **`external_kanban_ref`** (`20260527120000_...:880`).
- Nota operacional real: **no PROJUDI a data de audiência é marcada já no momento do protocolo** (agente `esp_protocolo_projudi`, `20260527120000_...:681-693`) — o protocolo pode **produzir** um dado de audiência.

## 2. Contexto — onde o protocolo entra no fluxo

```
atendimento → confeccao → revisao → [PROTOCOLO] → audiencia → execucao → ...
                             │            │
                   aprovação humana   ponto de bloqueio
                   (ver §2.1)         documental (ver §2.2)
```

### 2.1 Aprovação humana antes do protocolo (o "8.2")
> O repositório **não usa a numeração literal "8.2"**; a "aprovação humana" do card 8.2 mapeia para **três mecanismos já existentes** que precedem/condicionam o protocolo:

| Mecanismo | Onde | Papel na precedência do protocolo |
|---|---|---|
| **Etapa `revisao`** (dona: `socio`) | `kanban_next_stage`/`kanban_stage_owner_role` | Revisão humana da peça imediatamente antes de `protocolo`. |
| **Validação de tarefa (V18)** — `awaiting_validation` | `20260603120000_v18_task_validation.sql`; RPC `validate_user_task`; UI `src/pages/ValidationQueue.tsx` | Quando `task_types.requires_validation = true`, "concluir" desvia para `awaiting_validation`; aprovar → `completed`, rejeitar → `in_progress`. *(Os task_types de protocolo hoje têm `requires_validation = false` — ver decisão D-1 em §8.)* |
| **Gate documental** | `kanban_add_task_to_board` (`20260707181458_gate_documental_e_auto_liberacao.sql:46-62`) | Levanta exceção *"Gate documental: faltam documentos obrigatórios validados"* se algum documento `required` do cliente não estiver `validado`. **O protocolo é o ponto onde o sistema efetivamente bloqueia por falta de documento.** |

**Premissa da spec:** a peça só chega ao protocolo **após revisão/aprovação humana** e **com o gate documental satisfeito**. O protocolo não reabre nem contorna esses gates.

### 2.2 Ponto de bloqueio
Confirmado em `docs/superpowers/specs/2026-07-09-trilha-c-checklist-chat-design.md:6,13`: *"Bloqueio ocorre no protocolo (gate existente), não na entrada."* O protocolo herda esse contrato — se o gate não passou, a peça não é protocolável.

## 3. Entradas do protocolo

A etapa recebe um **pacote de protocolo** composto por:

| # | Entrada | Origem verificada | Observação |
|---|---|---|---|
| 3.1 | **Peça aprovada** (petição/manifestação/acordo) | Gerada por LLM e materializada em DOCX (`src/lib/bacellarDocx.ts`) e PDF (`src/lib/messageToPdf.ts`), exibida/baixada em `src/components/juris-cloud/PecaModal.tsx` | Deve estar na versão **revisada/aprovada** (§2.1), não em rascunho. |
| 3.2 | **Anexos / documentos do cliente** | `client_documents` (status `validado`), bucket privado `client-documents` | Só documentos com `status = 'validado'` são elegíveis (é o que o gate exige). |
| 3.3 | **Metadados do processo** | `processes` + `user_task` (via `external_kanban_ref`, `client_id`, `task_type`) | Comarca/vara/tribunal determinam o sistema externo de destino (PJe/PROJUDI/eproc/ProJuris "conforme comarca"). |
| 3.4 | **Tipo de protocolo** | `task_types.code` ∈ {`protocolar_peca`, `emendar_inicial`, `protocolar_acordo`} | Define prazo (SLA) e semântica (peça inicial × emenda × acordo). |
| 3.5 | **Prazo / data-fatal** | `user_task` (deadline/data_fatal) | `protocolar_peca`/`protocolar_acordo` têm SLA de 8h; acordo é "mesmo dia". |

**Contrato mínimo de entrada (o que a automação/assistente pode assumir como presente):** peça aprovada (3.1) + ao menos os documentos `validado` exigidos pelo gate (3.2) + processo com comarca/tribunal resolvidos (3.3) + `task_type` (3.4). Ausência de qualquer um = **falha de pré-condição** (ver F-0 na §4), tratada **antes** de tocar o sistema externo.

## 4. Modos de falha do sistema externo (catálogo)

Cada modo especifica **detecção**, **mensagem** e **ação**. Ações padronizadas:
- **RETRY** = reprocessável automaticamente (backoff), sem intervenção.
- **FILA** = enfileirar para nova tentativa quando a condição externa mudar (ex.: tribunal voltar).
- **ESCALAR** = exige humano (protocolador/advogado); nunca decidir sozinho.
- **BLOQUEAR** = pré-condição interna não satisfeita; não chega a tentar o protocolo.

| ID | Modo de falha | Detecção | Mensagem (ao operador) | Ação |
|---|---|---|---|---|
| **F-0** | **Pré-condição interna ausente** (peça não aprovada, doc obrigatório faltando, comarca/tribunal indefinidos) | Validação local antes de submeter | "Protocolo não iniciado: {pré-condição} pendente." | **BLOQUEAR** (volta ao gate/revisão) |
| **F-1** | **Formato/tamanho de arquivo inválido** (PDF/A exigido, limite de MB, assinatura no formato errado) | Regra local + rejeição do sistema | "Arquivo rejeitado: {motivo de formato}." | ESCALAR (regerar peça) — nunca "consertar" silenciosamente |
| **F-2** | **Certificado digital** (expirado, ausente, sem poderes/procuração) | Erro de autenticação do sistema externo | "Falha de certificado: {detalhe}." | ESCALAR (humano com token/certificado) |
| **F-3** | **Indisponibilidade do tribunal** (sistema fora do ar, timeout, manutenção) | Timeout / HTTP 5xx / página de manutenção | "Tribunal indisponível — reagendado." | **FILA** + RETRY com backoff; alertar se persistir além do SLA |
| **F-4** | **Erro de autuação** (classe/assunto inválido, campos obrigatórios do sistema) | Validação de campos do sistema externo | "Autuação recusada: {campo}." | ESCALAR (revisar metadados) |
| **F-5** | **Competência/vara errada** (comarca/vara/órgão incorretos) | Recusa por incompetência / seleção inválida | "Competência incorreta: {órgão}." | ESCALAR (advogado redefine competência) — **decisão jurídica, nunca do LLM** |
| **F-6** | **Peça fora do padrão** (documento não confere com a classe, peça inicial × emenda trocadas) | Validação semântica + recusa | "Peça incompatível com {classe/tipo}." | ESCALAR |
| **F-7** | **Prazo/data-fatal** (janela de protocolo expirando ou expirada) | Comparação com deadline/data-fatal do `user_task` | "Prazo em risco/vencido: {data}." | ESCALAR **imediato** (prioridade máxima); registrar tentativa e horário |
| **F-8** | **Duplicidade** (peça já protocolada, número já existente) | Retorno de duplicidade do sistema | "Possível duplicidade: {protocolo existente}." | ESCALAR (não reprotocolar automaticamente) |
| **F-9** | **Sucesso com efeito colateral** (protocolo aceito **e** data de audiência devolvida — caso PROJUDI) | Resposta de sucesso contém dados de audiência | "Protocolado; audiência marcada em {data}." | Registrar sucesso **e** propagar a audiência ao módulo de audiências (não perder o dado) |

**Regras transversais do catálogo:**
1. **Toda tentativa é registrada** (§6), inclusive as que falham em F-0/BLOQUEAR.
2. **RETRY é limitado** (nº máximo + backoff): esgotado, converte-se em ESCALAR — nunca loop infinito.
3. **Nenhuma ação destrutiva silenciosa.** Reprotocolar, cancelar ou substituir peça exige decisão humana.
4. **Idempotência:** antes de qualquer submissão, checar se já há protocolo bem-sucedido para o mesmo `(processo, peça, tipo)` — evita F-8 auto-infligido em retentativas.

## 5. Requisitos de LLM para a etapa

### 5.1 Tier de modelo
- **Tier forte / baixa alucinação** para qualquer julgamento textual da etapa (conferência peça × classe, leitura de mensagens de erro do tribunal, extração de dados de retorno). Alinhar ao **mesmo padrão de governança de PII** já usado em resumo/transcrição de atendimento: **OpenAI-direct (sem OpenRouter)** quando houver dado pessoal em contexto.
- **Justificativa:** protocolo é ato com efeito jurídico e prazo; erro de leitura/decisão tem custo alto (perda de prazo, autuação errada). O custo do modelo é irrelevante frente ao custo de um protocolo errado.

### 5.2 Restrição dura anti-alucinação
- **O LLM não inventa dado de protocolo.** Número de processo, classe, assunto, comarca, vara, partes, datas: **só** dos metadados de entrada (§3) ou do retorno literal do sistema externo. Campo ausente ⇒ **F-0/ESCALAR**, nunca preenchimento inferido.
- Reaproveitar o padrão já existente de validação mecânica anti-alucinação de citações/súmulas por tribunal (`supabase/functions/chat-orchestrator/mechanicalValidator.ts`) como referência de rigor.

### 5.3 O que o LLM PODE decidir vs NÃO PODE
| PODE (assistir) | NÃO PODE (decisão humana obrigatória) |
|---|---|
| Classificar a mensagem de erro do tribunal num dos modos F-1…F-9 | Redefinir competência/vara (F-5) |
| Extrair número de protocolo/data de audiência do retorno (F-9) | Reprotocolar após duplicidade (F-8) |
| Sinalizar prazo em risco (F-7) e priorizar | Alterar o conteúdo da peça aprovada |
| Sugerir a ação padronizada (RETRY/FILA/ESCALAR) | Confirmar submissão irreversível ao tribunal sem aprovação |
| Redigir a mensagem ao operador | Assinar/usar certificado digital (F-2) |

Toda saída de LLM que dispare ação externa passa pelo padrão **propor → confirmar** (equivalente a `proposeAction → ActionCard → confirm`), com registro em auditoria.

## 6. Requisitos de auditoria e log

**Modelo de referência (já em produção):** `task_audit_log` + trigger `kanban_audit_user_task` (`20260617160000_kanban_sp5_timesheet_audit.sql`) e `client_document_events` com `details jsonb` e FK `ON DELETE SET NULL` (`20260707135855_...:66-155`). A auditoria de protocolo deve seguir o **mesmo padrão** (append-only, sobrevive à exclusão do registro-pai).

### 6.1 Campos por tentativa de protocolo
| Campo | Descrição |
|---|---|
| `attempt_id` | Identificador único da tentativa |
| `user_task_id` / `processo_ref` | Vínculo com a tarefa/processo (via `external_kanban_ref` quando aplicável) |
| `tipo` | `protocolar_peca` / `emendar_inicial` / `protocolar_acordo` |
| `input_snapshot` | Referência imutável ao pacote de entrada (peça + hash dos anexos + metadados) — **referência/hash, não cópia de PII em claro** |
| `sistema_externo` | PJe / PROJUDI / eproc / ProJuris + órgão/comarca |
| `actor` | Quem disparou (humano) e, se houver, qual agente/modelo assistiu |
| `decisao` | Modo F-x classificado + ação (RETRY/FILA/ESCALAR/BLOQUEAR) |
| `resultado` | sucesso / falha + código e mensagem literal do sistema externo |
| `protocolo_externo` | Número/recibo devolvido em caso de sucesso |
| `efeitos` | Dados colaterais (ex.: data de audiência do PROJUDI — F-9) |
| `at` | Timestamp (crítico para F-7/prazo) |

### 6.2 Retenção e rollback
- **Retenção:** log de protocolo é **append-only** e retido pelo prazo legal do processo (não expira com o encerramento da tarefa). Segue `ON DELETE SET NULL` para sobreviver à exclusão da tarefa/documento.
- **Rollback:** protocolo aceito pelo tribunal é **irreversível pelo sistema** — não há "desfazer" técnico. O que se registra é a **necessidade de ato jurídico** (desistência/retificação), sempre humano. Toda "reversão" é um **novo evento auditado**, nunca edição/exclusão do evento original.
- **PII:** o snapshot guarda **referências/hashes**, não conteúdo pessoal em claro, coerente com a fase de criptografia de PII do projeto.

## 7. Dependência externa (fora de escopo desta fase)

| Item | Situação verificada |
|---|---|
| **Dono** | RPA / Integração (V2) — **outro dono**, não esta spec/fase. |
| **Estado hoje** | Só **placeholders**: agentes `esp_protocolo_pje`, `esp_protocolo_projudi`, `esp_protocolo_projuris` (`20260527120000_...:681-693`); campos de RPA "prontos, sem lógica" no módulo de audiências (`20260710215335_create_audiencias_module.sql:44-51`, `origem 'manual'|'automatica'`); gateway REST genérico `integration-api` (`docs/API_INTEGRACAO_EXTERNA.md`). **Não há integração real de protocolo implementada.** |
| **Protocolo real hoje** | **Ação humana** — o Protocolador opera o sistema do tribunal manualmente. |

### 7.1 Contrato que esta spec fixa para a automação futura
Quando a automação existir, ela deve **consumir** o contrato de entrada (§3) e **produzir** o contrato de saída (§4/§6):

- **Entrada (o que a automação recebe):** pacote de protocolo de §3 (peça aprovada + anexos `validado` + metadados de processo + tipo + prazo).
- **Saída (o que a automação devolve):** um dos resultados de §4 — sucesso (com `protocolo_externo` e eventuais `efeitos` como a audiência do PROJUDI) **ou** um modo de falha F-1…F-8 classificado, sempre com a **mensagem literal** do sistema externo, gravado conforme §6.
- **Garantias exigidas:** idempotência (§4.4), retentativa limitada (§4.2), nenhuma ação irreversível sem confirmação humana (§5.3), auditoria append-only (§6).

## 8. Critérios de aceite da spec

Esta **especificação** (não código) é aceita quando:

1. **§4 completo e validado por Ryan + Rodrigo:** o catálogo de modos de falha cobre os erros reais do dia a dia do escritório (formato, certificado, indisponibilidade, autuação, competência, peça fora do padrão, prazo, duplicidade) e cada linha tem detecção + mensagem + ação padronizada.
2. **§5 fixa o tier de LLM e a fronteira PODE/NÃO PODE**, com a restrição anti-alucinação e o alinhamento à governança de PII OpenAI-direct.
3. **§6 define campos, retenção e semântica de rollback**, reusando o padrão `task_audit_log`/`client_document_events`.
4. **§7 deixa explícito** que a automação de protocolo é dependência de outro dono (V2) e define o contrato de entrada/saída que essa automação deverá honrar.
5. **Decisões abertas resolvidas** (abaixo).
