**Data:** 02/07/2026 · **Branch base:** main
**Método:** 3 camadas por item (UI + código + banco via Supabase MCP). Regra cardinal respeitada: nada é "Implementado" sem evidência de banco/código; a tela dizer "sucesso" não é prova.

## Anexo D — Re-escopo já identificado (leva ao 0.3 Gap Analysis)

**Fase 3 (Clientes) muda de natureza — CONFIRMADO.** Os campos já são **capturados e persistidos** (50 colunas, 34 clientes reais); o problema é de **exibição**, não de captura.

- **3.2 / 3.3 / 3.4** viram **"expor o que existe"** (mais leve): a tela mostra ~10 de 50 colunas. Bancários, filiação, dados PJ e `gov_br_profile` são coletados e não aparecem em lugar nenhum.
- **Editar cliente** (dentro da Fase 3) é **Não implementado** — aqui é **construir**, não só expor (não há `.update()` em `clients`).
- **3.5 (Gov.br/segurança)** é **reclassificado**: não há "senha gov.br" a descriptografar. O crítico é **blindar dado sensível já em trânsito sem proteção** (CPF/RG/conta/PIX/`gov_br_profile` em plaintext, RLS só de linha por papel). Sobe de prioridade como **Hardening** (R-2), não como bug de exibição.
- **Documentos do cliente**: upload real, mas **sem status/origem/log** — ajuste incremental.

Impacto de cronograma: parte da Fase 3 encolhe (exibição), mas ganha um card novo de **Hardening de dados de cliente** e um card de **CRUD de edição**. Registrar no 0.3.
